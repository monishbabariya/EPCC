## BLOCK 3 — Data Architecture

### 3a. Entities

| Entity | Description | Cardinality |
|--------|-------------|-------------|
| `EVMSnapshot` | Point-in-time EVM calculation per WBS node per reporting period | Many per WBS node |
| `PackageEVMRollup` | Package-level EVM aggregation — rolled up from WBS snapshots | Many per package per period |
| `ProjectEVMSummary` | Project-level EVM summary — rolled up from all packages | 1 per project per period |
| `PortfolioEVMSummary` | Portfolio-level EVM — absolute aggregation across all active projects | 1 per portfolio per period |
| `EACConfig` | Designates which EAC method feeds M10 Command per project | 1 per project |
| `KPIEntry` | Weekly/periodic KPI register — health scores and RAG per project | Many per project |
| `TrendAnalysis` | Statistical trend output per metric per project | 1 per metric per project per period |
| `PredictiveAlert` | Forward-looking breach prediction — generated by trend engine | Many per project |
| `RecalcQueue` | Tracks all pending recalculation jobs — source change → queue → execution | Many per project |
| `TrendConfig` | Configures which metrics have trend analysis enabled per project | 1 per project |
| `ProjectRecalcLock` | **(NEW v3.0)** Manages cascade isolation — prevents concurrent recalculation conflicts | 1 per project (active record) |
| `RecalcConflictLog` | **(NEW v3.0)** Immutable log of every lock collision event for system monitoring | Many per project |
| `BACIntegrityLedger` | **(NEW v3.0)** Audit trail of every BAC change — VO materialisation, baseline revision, correction | Many per project |

**Rules:**
- One `ProjectRecalcLock` record per project. `lock_active = false` when idle.
- Lock acquisition and release are atomic DB operations (PostgreSQL SELECT FOR UPDATE)
- Maximum lock duration: 120 seconds. If exceeded → system alert + force-release + RecalcConflictLog

### New and Updated Rules — v3.0

| Rule ID | Trigger | Logic | Output | Speed Tier |
|---------|---------|-------|--------|------------|
| BR-07-022 | RecalcQueue job starts | Before any calculation: check `ProjectRecalcLock.lock_active` for this project | If lock free → acquire lock → proceed. If lock held by STANDARD → supersede, defer STANDARD job, proceed. If lock held by IMMEDIATE → set new job to `Pending_Lock_Release`. Log collision in RecalcConflictLog. | 🔴 Real-time |
| BR-07-023 | RecalcQueue job completes | Release `ProjectRecalcLock`. Evaluate all Deferred jobs for same project. | If deferred job trigger data superseded by completed cascade → status = Collapsed. If new trigger data exists → create single consolidated STANDARD job. Write `lock_duration_ms`. | 🔴 Real-time |
| BR-07-024 | ProjectRecalcLock held > 120 seconds | System detects lock timeout | Force-release lock. Set job_status = Failed. System Admin alert generated. RecalcConflictLog entry: `conflict_type = LOCK_TIMEOUT`. Retry job immediately with fresh lock attempt. | 🔴 Real-time |
| BR-07-025 | RecalcQueue job starts — BAC integrity check | Before EVM calculation: read `ProjectEVMSummary.bac_integrity_status` | If `Confirmed` → proceed normally (bac_check_status = Passed). If `Stale_Pending_VO` → proceed with CPI/SPI/EV/AC but set `eac_cpi_suspended = true`, `eac_spi_cpi_suspended = true`, `eac_etc_reestimate_suspended = true` on all affected EVMSnapshots. bac_check_status = Suspended_BAC_Stale. | 🔴 Real-time |
| BR-07-026 | EAC suspension active | EAC fields null in EVMSnapshot | `eac_suspension_note` = "EAC suspended: VO {vo_code} materialisation pending since {date}. Resolve in M05 Variation Orders." Propagated to M10 as dashboard badge. | 🔴 Real-time |
| BR-07-027 | EVMSnapshot computation — Subjective_Estimate entry | `ProgressEntry.progress_approval_status` for this WBS node | If `Approved` → use `pct_complete_reported` normally. ev_confidence = Low. If `Draft` → use fallback EV from last Approved ProgressEntry for same WBS node. If no prior Approved entry → fallback = 0.0. Set ev_approval_pending = true, ev_fallback_used = true, ev_confidence = Fallback. pct_complete_draft stored for reference only. | 🔴 Real-time |
| BR-07-028 | EVMSnapshot computed with ev_fallback_used = true | Fallback EV used for WBS node | Add this WBS node to `ProjectEVMSummary.ev_approval_pending_count`. Add fallback EV amount to `ProjectEVMSummary.total_ev_fallback_amount`. | 🔴 Real-time |
| BR-07-029 | Total ev_approval_pending WBS nodes > 10% of active WBS count | Check run after every calculation | Decision Queue item: `trigger_type = HIGH_EV_APPROVAL_PENDING`. Severity = MEDIUM. Owner = Project Director. SLA = 48hr. Context: pending_count, pct_of_total. | 🟡 2-4hr |
| BR-07-030 | `etc_is_stale` becomes true | After period-end check | Create `DecisionQueueItem`: trigger_type = `ETC_STALE`, severity = `HIGH` if EAC_ETC_Reestimate is designated method, else `MEDIUM`. Owner = Planning_Engineer. SLA = 72hr. Set `etc_stale_decision_queue_id`. Single alert per staleness event (check `etc_stale_alert_sent` before creating). | 🟢 24hr |
| BR-07-031 | ETC stale Decision Queue item SLA breached at 72hr | SLA escalation | Re-assign to Project Director. SLA extension = 72hr. | 🟡 2-4hr |
| BR-07-032 | ETC stale + EAC_ETC_Reestimate is designated method + stale > 1 period | Combined check | M10 project card receives `DATA_CONFIDENCE = WARNING` badge. Badge label: "EAC based on stale estimate — {N} periods since update". Cleared when new ETC re-estimate submitted. | 🟢 24hr |
| BR-07-033 | `tcpi_feasibility` = `Unlikely` (TCPI > 1.20) | Detected in EVMSnapshot | Send signal to M08: `tcpi_infeasibility_alert` with project_id, tcpi value, reporting_period. M08 creates `Gate_Health_Review` Decision Queue item (see M08 v2.1 BR-08-XXX). M10 receives alert for portfolio war room. | 🟡 2-4hr |
| BR-07-034 | BAC materialisation completes (VOBOQMaterialisation → Completed, received from M05) | M02 sends updated BAC to M07 | `ProjectEVMSummary.bac_integrity_status` → Confirmed. Create STANDARD RecalcQueue job. All suspended EAC fields will be re-computed in next recalc run. BACIntegrityLedger entry written. | 🔴 Real-time |
| BR-07-035 | Total Subjective_Estimate EV > 15% of project total EV | Check run after every calculation | Alert to PMO Director: "High proportion of subjective EV ({pct}% of total). Consider converting high-value activities to measurable methods." Informational — not a block. Single alert per period. | 🟢 24hr |

## BLOCK 9 — Explicit Exclusions (v3.0 — additions in bold)

```
This module does NOT:
────────────────────────────────────────────────────────────────────────
[ ] Measure or capture % complete                              → M04
[ ] Generate or own Planned Value (PV)                        → M03
[ ] Record or own Actual Cost (AC) transactions               → M06
[ ] Define or store BOQ items or BAC structure                → M02
[ ] Run Monte Carlo simulation                                → M05
[ ] Approve stage gates                                       → M08
[ ] Track regulatory compliance                               → M09
[ ] Process billing or cashflow                               → M06
[ ] Re-apply progress measurement methods                     → M04
[ ] Independently compute parent EV from children             → Architectural violation
[ ] Accept manual EV or % complete overrides                  → Never — read-only consumer
[ ] Approve or reject Subjective_Estimate progress entries    → M04 (ProgressEntryApproval)
[ ] Execute or manage the VO materialisation workflow         → M05 + M02
[ ] Decide whether to release lock on cascade failure         → RecalcQueue service
[ ] Write directly to M08 gate records                        → Sends signal only; M08 owns gate logic
```
